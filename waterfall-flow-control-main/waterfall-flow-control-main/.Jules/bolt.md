## 2025-02-18 - [Optimization: Split Filtering and Sorting in `useTransactionFilters`]
**Learning:** In list-filtering hooks, combining filtering (expensive string matching) and sorting (cheap O(N log N)) in a single `useMemo` is a performance anti-pattern. Every time the sort order changes (a frequent UI action), the expensive filter logic re-runs redundantly.
**Action:** Always split `useMemo` into two stages: 1) `filteredItems` (depends on query), 2) `sortedItems` (depends on `filteredItems` + sort criteria). This is essentially a "selector" pattern. Additionally, exposing the intermediate `filteredItems` allows other consumers (like Charts/Stats) to ignore sort order entirely, preventing massive re-renders of heavy visualization components when the user just wants to reorder the table.
