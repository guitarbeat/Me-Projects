## 2024-05-24 - `toISOString` Performance Overhead
**Learning:** `new Date(...).toISOString().split('T')[0]` is an expensive operation inside loops, particularly for rendering calendar views where it runs hundreds of times per render. It's approximately 10-100x slower than manually extracting and concatenating year/month/day components. Furthermore, it inherently operates in UTC, which can introduce subtle "off-by-one-day" timezone shift bugs when working with purely local dates (e.g. `new Date(2023, 5, 15).toISOString()` returning `2023-06-14T...` depending on system timezone).
**Action:** When repeatedly formatting local dates in hot paths (like loops or frequent renders), prefer manual string concatenation (e.g. ``${year}-${m < 10 ? '0' + m : m}-${day < 10 ? '0' + day : day}``) instead of instantiating Dates and calling `.toISOString()`. Use utility functions like `formatDateComponents` and `toLocalDateString` to encapsulate this logic.
